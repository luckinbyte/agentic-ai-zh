# 第 19 章 智能体设计模式

构建高效的智能体(agent)需要的不仅仅是一个强大的模型和一组工具。架构——即如何编排 LLM、如何分解任务、控制流如何在组件之间流动——决定了智能体是否可靠、可调试且具备成本效益。本章介绍源自 Anthropic、OpenAI、Google 及开源社区在生产部署中沉淀出的若干经典设计模式(design pattern)。

## 何时使用智能体,何时使用工作流

并非每项任务都需要自主智能体。关键区别在于:

- **工作流(Workflow)**:预定义控制流,在特定步骤调用 LLM。可预测、可测试、更廉价。适用于任务结构已知的场景。
- **智能体(Agent)**:由 LLM 动态决定下一步做什么。灵活,能处理新颖情形。适用于任务需要自适应决策的场景。

请从工作流开始。只有当任务确实需要动态路由或开放式探索时,才升级到智能体。

## 19.1 工作流模式

这些模式改编自 Anthropic 对智能体基本构建单元的分类法 [342],它们在预定义的控制流中使用 LLM。由系统(而非模型)决定执行顺序。

### 19.1.1 提示链(Prompt Chaining)

最简单的模式:将一个复杂任务拆分为固定的 LLM 调用序列,把一次调用的结果作为上下文传递给下一次调用。步骤之间的校验门(gate)能在错误向下游传播之前及早捕获错误。

![图 19.1:带质量门的提示链。每一步都是一次独立的 LLM 调用,门可以是基于 LLM 的,也可以是程序化的。](images/part-v-agentic-ai/agent-design-patterns/agent-design-patterns-p369-01.png)

**适用场景**:天然顺序化的任务——内容生成、数据转换、多阶段分析。

**关键优势**:每一步可以使用不同的提示(prompt)、模型或温度(temperature)。中间结果可检视、可调试。

### 19.1.2 路由(Routing)

一个分类器(LLM 或传统分类器)检视输入并将其分发给专门的处理器。

![图 19.2:路由模式:输入先被分类一次,然后交由专家处理器处理。](images/part-v-agentic-ai/agent-design-patterns/agent-design-patterns-p370-02.png)

**适用场景**:具有不同最优提示、工具或模型的多种任务类型。如客户支持分诊(triage)、多模态输入处理。

### 19.1.3 并行化(Parallelization)

多个 LLM 调用并发执行,由一个程序化层合并它们的输出。它衍生出两个子模式:

- **切分(Sectioning / 扇出 fan-out)**:将输入划分为不相交的分块并各自独立处理——例如同时在一个代码库上运行安全检查、性能检查和风格检查。
- **投票(Voting / 冗余 redundancy)**:以不同的随机种子或温度对同一提示(prompt)发起 N 次,然后通过多数投票 [343]、奖励模型打分或 LLM-as-judge 选出最佳结果。

**并行化示例:代码审查**

1. 并行调用:安全审查 ∥ 性能审查 ∥ 风格审查
2. 聚合:合并所有发现,去重,按严重程度排序

延迟为 $\max(\text{各次调用})$,而非各次调用之和 $\sum(\text{各次调用})$。

### 19.1.4 编排者-工人(Orchestrator-Workers)

这里由 LLM 自身决定如何拆分工作。一个编排者(orchestrator)模型分析任务,产出子任务计划,将每个子任务分发给工人(worker)LLM(可能使用不同的提示或工具),最终把它们的输出合并为一个连贯的结果。它与并行化的关键区别在于:分解逻辑是由模型生成的,而非硬编码。

**适用场景**:子任务的数量和性质在设计时无法预先枚举的开放式问题——例如"重构这个代码库"需要先理解依赖图,然后才能决定修改哪些文件。

### 19.1.5 评估者-优化器(Evaluator-Optimizer)

这是一个双模型反馈回路 [240]:生成器(generator)产出候选输出,同时一个独立的评估器(evaluator)根据明确的准则对其打分。如果分数低于阈值,就把评估器的批评意见追加到生成器的上下文中,如此循环往复,直到达到质量门槛或重试预算耗尽。

**适用场景**:具有明确质量准则的任务——必须通过测试的代码、必须保留原意的翻译、必须符合风格指南的写作。

![图 19.3:编排者-工人:由 LLM 决定如何分解任务,并综合各工人的结果。](images/part-v-agentic-ai/agent-design-patterns/agent-design-patterns-p371-03.png)

![图 19.4:评估者-优化器:无需训练的迭代精化。](images/part-v-agentic-ai/agent-design-patterns/agent-design-patterns-p371-04.png)

## 19.2 自主智能体模式

这些模式将执行流的控制权交给 LLM 自身。

### 19.2.1 ReAct(Reason + Act)

这是最基础的智能体模式 [127]。LLM 在思考(内部推理)、行动(工具调用)和观察(处理结果)之间交替循环,直到产出最终答案。

**ReAct 实现要点**

- **草稿本(Scratchpad)**:"思考(Thought)"步骤会被记录但不对用户展示。
- **工具解析**:运行框架(harness)从模型输出中提取结构化的工具调用。
- **最大迭代次数**:始终为循环设上限(典型为 10–25 次迭代)。
- **终止条件**:模型输出一个特殊动作(如 `final_answer`),或检测不到工具调用。

### 19.2.2 规划智能体(Planning Agents)

智能体在执行之前生成显式的计划,并可在执行过程中修订计划 [126]。

**规划智能体示例:研究报告撰写**

用户请求:"撰写一份 2 页报告,比较用于时间序列预测的 Transformer 架构。"

步骤 1 —— 计划生成(单次 LLM 调用):

```python
plan = [
    {"id": 1, "task": "Search for recent "
              "transformer-based time-series models (2023-2025)",
     "tool": "search_web", "deps": []},
    {"id": 2, "task": "Read top 5 papers, extract key methods",
     "tool": "read_papers", "deps": [1]},
    {"id": 3, "task": "Build comparison table (architecture, "
              "dataset, metrics)",
     "tool": "none", "deps": [2]},
    {"id": 4, "task": "Write introduction + methodology section",
     "tool": "none", "deps": [2]},
    {"id": 5, "task": "Write results + conclusion",
     "tool": "none", "deps": [3, 4]},
    {"id": 6, "task": "Review and polish final report",
     "tool": "none", "deps": [5]},
]
```

步骤 2 —— 带自适应重规划的执行:智能体按依赖顺序执行各步骤。在步骤 1 之后,搜索仅返回 3 篇相关论文。智能体重新规划:增加一个子步骤,将搜索范围扩展到相邻领域(例如 PatchTST、iTransformer)。修订后的计划从步骤 2 开始,使用扩展后的语料继续。

**关键洞见**:计划是一份"活的"文档——它提供结构,但能根据观察进行调整。运行框架(harness)将依赖关系作为有向无环图(DAG)进行跟踪,只执行那些前驱步骤已完成的步骤。

**表 19.1:规划策略对比**

| 策略 | 重规划时机 | 特征 |
|---|---|---|
| 先规划后执行(Plan-then-Execute) | 从不 | 简单;对意外结果脆弱 |
| 自适应(Adaptive) | 失败时 | 仅在步骤失败时重规划;成本适中 |
| 持续(Continuous) | 每一步 | 每次观察后全面重评估;昂贵但稳健 |
| 分层(Hierarchical) | 子计划完成时 | 高层计划固定;子计划动态生成 |

### 19.2.3 反思与自我批评(Reflection and Self-Critique)

智能体暂停下来评估自己的轨迹并修正路线:

1. **输出校验**:"这是对的吗?我漏掉了什么吗?"
2. **轨迹回顾**:回顾最近 k 步,识别错误或低效之处。
3. **策略修订**:重新审视整体方法("我是否在解决正确的问题?")。

**Reflexion:从失败中学习**

Reflexion 模式 [224] 维护一份持久的"反思记忆"。每次失败的尝试之后,智能体会写下一段自然语言反思("我失败了,因为我没检查边界情况")。在下一次尝试时,这些反思会被纳入提示——从而在不更新权重的情况下,实现跨回合(episode)的学习。

### 19.2.4 工具使用模式(Tool-Use Patterns)

智能体调用工具的方式显著影响其可靠性、延迟和成本。目前已经形成了五种经典模式 [332]:

**单轮工具使用(Single-Turn Tool Use)。**
最简单的模式:模型发起一次工具调用,接收结果,然后产出最终答案。足以应对事实查询、单位换算或单次 API 查询。运行框架(harness)只做恰好两次 LLM 调用(一次决定使用哪个工具,一次综合结果)。

**多工具(并行,Multi-Tool)。**
现代 API(OpenAI、Anthropic)允许模型在单次响应中请求多个工具调用。运行框架并发执行这些调用,并将所有结果一起返回。这极大降低了那些需要从多个来源获取独立信息的任务的延迟——例如同时获取股价、天气和日历。关键约束是:这些工具必须是相互独立的(没有任何工具的输出需要作为另一个工具的输入)。

**顺序式(流水线,Sequential)。**
每个工具的输出流入下一个工具的输入,形成一条数据流水线。模型根据前一次结果决定下一个工具。常见于研究类工作流:search → fetch_page → extract_data → analyze。运行框架必须跟踪不断增长的上下文,并可能需要对中间结果进行摘要以控制预算。

**嵌套式(智能体即工具,Nested / Agent-as-Tool)。**
一次工具调用会唤起一个完全独立的智能体——它拥有自己的提示、工具和上下文。父智能体把子智能体当作一个黑箱函数。这实现了专业化:一个研究智能体可以把代码执行委托给一个编码智能体,后者拥有沙箱(sandbox)和测试运行器的访问权限。Swarm 模式 [336] 通过专门化智能体之间的移交(handoff)对此进行了推广。

**降级(优雅退化,Fallback / Graceful Degradation)。**
运行框架按优先级顺序尝试工具:如果首选工具失败(超时、限流、API 错误),就自动降级到备选工具。模型无需感知降级逻辑——运行框架透明地处理它。例如:主搜索 API → 备用搜索 → 缓存结果 → 通知模型搜索不可用。

**表 19.2:工具调用模式**

| 模式 | 描述 | 示例 |
|---|---|---|
| 单轮(Single-turn) | 每次 LLM 响应一次工具调用 | 带搜索的简单问答 |
| 多工具(Multi-tool) | 一次响应中多个并行工具调用 | 搜索 + 计算 + 格式化 |
| 顺序式(Sequential) | 工具输出流入下一个工具调用 | 搜索 → 读取 → 提取 |
| 嵌套式(Nested) | 工具调用触发另一个智能体 | 代码智能体调用测试运行器 |
| 降级(Fallback) | 首选工具失败时尝试备选方案 | API → 抓取 → 缓存 |

## 19.3 设计原则

以下原则提炼自 Anthropic 的《构建高效智能体指南》[342],适用于所有模式:

1. **保持简单。** 使用能解决问题的最简架构。仅在确有必要时才增加复杂性。一个能解决问题的提示链,永远胜过一个"也许能解决"的多智能体系统。
2. **透明优于巧妙。** 每一步都应当可检视。避免隐藏状态或隐式推理。当智能体失败时,你需要理解原因——不透明的架构会让调试变得不可能。
3. **提供好用的工具。** 文档完善、类型清晰、错误信息明确的工具是力量倍增器。描述含糊的工具会被误用;具备精确 schema 与用法指引的工具会被正确选用。
4. **为失败做预案。** 每一次工具调用都可能失败。在运行框架层面构建重试逻辑、降级与优雅退化,使模型不必去推理基础设施故障。
5. **使用结构化输出。** 受约束的生成(JSON schema、函数调用)可防止解析失败。产出需要正则解析的自由文本的智能体是脆弱的;而产出经过校验的 JSON 的智能体是稳健的。
6. **用多样化输入测试。** 智能体行为比单轮对话更具多变性。同一提示在不同运行中可能产生不同的工具调用序列。要对抗性地测试,使用边界情况、含糊请求和畸形输入。

## 19.4 模式选择指南

选择正确的模式取决于三个因素:(1) 任务结构有多可预测,(2) 在延迟和成本上你能负担多少次 LLM 调用,(3) 质量是否需要迭代。将下表作为决策矩阵——从最简单的(最上方)开始,仅当更简单的模式确证失效时,才向下升级。

**表 19.3:何时使用各类智能体设计模式**

| 模式 | 复杂度 | LLM 调用次数 | 最适用场景 |
|---|---|---|---|
| 提示链(Prompt chaining) | 低 | N(固定) | 顺序任务、内容流水线 |
| 路由(Routing) | 低 | 1 + 1 | 多类型输入、分诊 |
| 并行化(Parallelization) | 低 | N(并行) | 独立子任务、投票 |
| 编排者-工人(Orchestrator-workers) | 中 | 可变 | 分解方式未知 |
| 评估者-优化器(Evaluator-optimizer) | 中 | 2–10(循环) | 质量至关重要的输出 |
| ReAct | 中 | 3–25(循环) | 通用工具使用、探索 |
| 规划智能体(Planning agent) | 高 | 5–50+ | 长周期、多步任务 |
| 反思(Reflection) | 高 | +50% 开销 | 首次尝试常失败的任务 |
| 多智能体(Multi-agent) | 高 | 多次 | 复杂领域、专业化分工 |

模式是可组合的:一个规划智能体可以在各个步骤内部使用提示链,在审查阶段内嵌评估者-优化器,并用路由将子任务分发给专家。真正的艺术,在于知道何时停止增加层次。
