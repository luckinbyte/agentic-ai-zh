# 第 22 章 智能体技能(Agent Skills)

随着智能体(agent)从单一的「提示 + 工具」系统演进为模块化架构,一个关键的设计问题随之浮现:智能体的能力应当如何被组织、发现与组合?答案日益收敛于「技能」(skill)这一概念——它是离散、可复用的行为单元,可以在无需重新训练的情况下被加载、组合与替换。

这一理念由 Voyager [228] 推广开来:它证明了在 Minecraft 中的 LLM 智能体能够积累一个不断增长的、可执行代码形式的技能库,每个技能都经过验证并存储以供日后复用。同样的原则也适用于生产环境中的智能体:技能将领域专长封装为可组合、可版本化的格式,其扩展能力远超任何单一提示所能承载的内容。技能常常封装 MCP 服务器(第 21 章)以实现工具访问,从而把技能抽象层与标准化工具层连接起来。

## 22.1 什么是技能?

技能(skill)是一个自包含的能力模块,它赋予智能体在特定领域或任务上的专长。与原始工具(只暴露单个函数)不同,技能涵盖以下内容:

- **系统提示增强**:注入到智能体上下文中的领域专属指令、约束与人设要素。
- **工具绑定**:技能所需的一个或多个工具(API、MCP 服务器、本地命令)。
- **知识**:智能体为正确执行该技能所需的参考资料、示例或少样本(few-shot)示范。
- **工作流逻辑**:引导智能体完成复杂任务的多步流程、决策树或条件分支。
- **护栏(guardrails)**:技能专属的安全约束、输出格式要求与校验规则。

| 概念 | 范畴 | 示例 |
|---|---|---|
| 工具(Tool) | 单次函数调用 | `web_search(query)` |
| 技能(Skill) | 一致的能力(提示 + 工具 + 知识) | 「研究分析师」(Research Analyst)技能 |
| 智能体(Agent) | 具备多种技能的自主实体 | 一个编码助手 |

> 工具是一把锤子。技能是懂得如何搭建一栋房屋的框架。智能体则是那位木匠——由他来决定使用哪些技能。

## 22.2 技能架构模式

### 22.2.1 静态技能加载(Static Skill Loading)

最简单的模式:在智能体初始化时依据配置加载技能。智能体始终可以访问其全部技能。

```python
# 伪代码 —— 框架无关的模式
agent = Agent(
    model="claude-sonnet-4-20250514",
    skills=["code-review", "documentation", "testing"],
    # 每个技能会为智能体添加提示、工具与知识
)
```

**优点**:简单、可预测、低延迟。

**缺点**:当技能未被使用时会浪费上下文窗口;无法扩展到数百个技能。

### 22.2.2 动态技能发现(Dynamic Skill Discovery)

智能体根据当前任务选择要激活哪些技能。一个技能路由器(skill router,通常是轻量级分类器或基于嵌入(embedding)的匹配器)决定相关性:

```python
# 伪代码 —— 框架无关的模式
relevant_skills = skill_router.match(
    user_request=message,
    available_skills=skill_registry,
    max_skills=3
)
agent.activate(relevant_skills)
```

**优点**:可扩展到大型技能库;上下文高效。

**缺点**:路由出错可能遗漏相关技能;会增加延迟。

### 22.2.3 层次化技能组合(Hierarchical Skill Composition)

技能可以依赖其他技能,从而形成一个有向无环图(DAG)。一个高层技能(例如「部署应用」)可能会调用若干子技能(「运行测试」「构建 Docker 镜像」「更新 DNS」):

- 技能显式声明其依赖项
- 编排器(orchestrator)在执行前解析依赖图
- 子技能可在多个父技能之间共享

## 22.3 案例研究:Anthropic 的智能体设计

Anthropic 的智能体架构方法 [342] 为生产环境中的技能化智能体设计提供了最清晰的阐述之一。其设计哲学强调:宁简勿繁,优先采用可组合的构建块而非单一庞大的框架。(这些模式在第 19 章中也曾从编排的角度加以讨论。)

### 22.3.1 核心原则

1. **从最简单的方案入手。** 在更简单的方法(单次 LLM 调用、检索 + 生成)被尝试并证明不足以胜任之前,不要急于采用智能体式模式。
2. **工作流与智能体之分。** Anthropic 区分了两类:
   - **工作流(Workflows)**:LLM 调用的预先定义编排——确定性控制流,仅在特定节点设置 LLM 步骤。更可预测、更易调试。
   - **智能体(Agents)**:LLM 动态决定下一步做什么——工具选择、迭代次数以及停止准则均由模型驱动。更灵活,但也更难控制。
3. **以增强型 LLM 作为原子单元。** 这一基本元素绝不是裸模型——它始终是一个与检索来源、可调用工具及持久化记忆捆绑在一起的模型。在实践中,这个复合单元正是一个装备了技能的模型。

### 22.3.2 构建块模式

Anthropic 给出了五种可组合的工作流模式,它们充当技能模板:

**表 22.1:Anthropic 的可组合智能体模式。**

| 模式 | 机制 | 适用场景 |
|---|---|---|
| 提示链(Prompt Chaining) | 顺序 LLM 调用,每一步的输出作为下一步的输入;步骤之间设置门控以校验中间结果 | 能够清晰分解的多步转换 |
| 路由(Routing) | 分类器或 LLM 根据任务类型将输入导向专门的处理器(技能) | 需要不同专长的、互不相同的任务类别 |
| 并行化(Parallelization) | 多个 LLM 调用同时运行——或为切分(拆分任务),或为投票(同一任务、聚合结果) | 相互独立的子任务;或通过共识获取置信度 |
| 编排器–工人(Orchestrator–Workers) | 中央 LLM 将任务拆分为子任务、委派给工人 LLM,再综合结果 | 子任务无法预先确定的复杂任务 |
| 评估器–优化器(Evaluator–Optimizer) | 一个 LLM 生成、另一个 LLM 评估;迭代直至达到质量阈值 | 具备明确质量标准的任务(代码、写作) |

### 22.3.3 增强型 LLM

在 Anthropic 的框架中,基本单元不是裸模型,而是增强型 LLM(augmented LLM):

$$\text{Augmented LLM} = \text{Model} + \text{Retrieval} + \text{Tools} + \text{Memory}$$

这与技能概念直接对应:每个技能配置的是模型在特定任务中可访问哪些检索来源、工具与记忆存储。技能的边界界定了模型在某次具体调用中能够看见什么、能够做什么。

### 22.3.4 实践启示

> **Anthropic 的关键洞察**
>
> 最有效的智能体并不是最复杂的那些。它们是拥有优良工具的简单循环:

```python
while not done:
    action = llm.decide(context, tools)
    result = execute(action)
    context.append(result)
    done = llm.should_stop(context)
```

智能性来自:(1)模型自身的能力,(2)工具描述的质量,(3)任务表述的清晰程度——而非精巧复杂的编排逻辑。技能为(2)和(3)提供了结构支撑。

源自 Anthropic 方法的设计建议:

- **保持智能体循环简单**:避免对控制流过度工程化。让模型来做决定。
- **在工具质量上投入**:详尽、无歧义的工具描述比复杂的路由逻辑更有价值。
- **使用结构化输出**:强制模型以可解析的格式(JSON、函数调用)输出决策——可减少技能执行错误。
- **内置恢复机制**:技能应当优雅地处理错误——使用不同参数重试、请求澄清,或升级交由人工处理。
- **限制每个技能的范围**:试图包揽一切的技能将一事无成。狭窄、定义明确的技能比宽泛的技能更易于组合。

## 22.4 技能生命周期(Skill Lifecycle)

1. **发现(Discovery)**:系统识别出有哪些技能可用(注册表、市场、本地定义)。
2. **选择(Selection)**:根据用户请求,匹配并加载相关技能。
3. **激活(Activation)**:将技能的提示、工具与知识注入到智能体的上下文中。
4. **执行(Execution)**:智能体使用技能的能力来完成任务的执行。
5. **停用(Deactivation)**:移除技能上下文,以释放上下文窗口供后续任务使用。
6. **学习(Learning)**:执行结果可用于更新技能的少样本示例,或对路由进行微调。

## 22.5 技能注册表与市场(Skill Registries and Marketplaces)

生产级技能系统需要相应的基础设施:

- **技能清单(skill manifest)**:一种结构化描述(名称、能力、所需工具、输入/输出模式),用以支持自动发现与路由。
- **版本控制**:技能会演进;智能体需要锁定特定版本以保证可复现性。
- **依赖解析**:技能可能需要特定的 MCP 服务器、API 密钥或其他技能。
- **权限模型**:并非所有智能体都应访问所有技能(出于安全、成本、能力边界的考虑)。
- **市场(marketplace)**:组织可以发布、分享并安装技能——类似于代码所用的包管理器。

### 技能清单示例

技能清单声明了编排器加载并调用一个技能所需的一切信息。目前尚不存在业界标准模式;下面给出一个示意性格式,它捕获了真实实现(Anthropic MCP、OpenAI 函数规格、LangChain 工具定义)中常见的字段:

```json
// 示意性模式 —— 并非某个特定 SDK 的格式
{
  "name": "code-review",
  "description": "Review code changes for bugs, style, and security issues",
  "version": "2.1.0",
  "requires": {
    "tools": ["file_read", "grep", "git_diff"],
    "mcp_servers": ["github"],
    "models": ["claude-sonnet-4-20250514"]
  },
  "input_schema": {
    "type": "object",
    "properties": {
      "repo": {"type": "string"},
      "pr_number": {"type": "integer"}
    }
  },
  "prompts": ["skills/code-review/system.md"],
  "knowledge": ["skills/code-review/style-guide.md"]
}
```

## 22.6 技能与微调的对比

一个自然的问题是:为什么要使用运行时技能注入,而不是对模型进行微调?

**表 22.2:就新增能力而言,技能(上下文内)与微调的对比。**

| 维度 | 技能(上下文内) | 微调 |
|---|---|---|
| 部署速度 | 即时 | 数小时至数天 |
| 灵活性 | 运行时可替换/组合 | 训练时即固定 |
| 上下文开销 | 占用上下文窗口 | 零运行时开销 |
| 深层行为改变 | 受上下文长度限制 | 深层参数化改变 |
| 多租户 | 不同用户配不同技能 | 所有用户共享同一模型 |
| 维护 | 更新文本文件即可 | 需在新数据上重新训练 |

实践中,两种方法是互补的:微调提供基础能力(指令遵循、工具使用格式、推理),而技能则在运行时于其之上叠加任务专属的专长。
